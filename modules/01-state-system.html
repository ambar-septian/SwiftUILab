Module 01

SwiftUI State System

Ownership, value semantics, and the observation model that decides when a view's body re-runs.

Concept

SwiftUI views are value types that are cheap and disposable. The struct you write is not the thing on screen — it's a description that SwiftUI re-evaluates whenever its inputs change. The entire "state system" exists to answer one question: where does the source of truth live, and who gets invalidated when it changes? Coming from UIKit, the mental flip is that you never hold a long-lived view object and mutate it; you mutate state, and the framework recomputes the view tree and diffs it.

@State — owned, value-typed source of truth

@State gives a view its own private, persistent storage that survives body re-execution. The struct is recreated constantly; the @State storage is allocated outside the struct (in SwiftUI-managed memory) and reconnected on each recreation by structural identity. Mutating it invalidates the view. Use it for value types the view owns — never share the same @State across views by copying.

@Binding — a reference into someone else's state

A Binding<T> is a read/write projection — a get/set pair pointing at storage owned elsewhere. $value creates one from @State. The child neither owns nor copies the value; writing through the binding mutates the owner and invalidates it. This is value-semantics two-way data flow without UIKit's delegate/target-action plumbing.

When bindings "break": if you write let copy = value in the child and mutate copy, you've left the binding — you're mutating a value-typed copy that goes nowhere. Always mutate through $/_value.wrappedValue. Likewise, a binding derived inside body with stale captured state can write back outdated values.

Custom Binding(get:set:)

When the stored representation and the presented representation differ — clamping, formatting, mapping optional↔non-optional, deriving from a collection — build a binding by hand. It's just a transform layer over storage:

let clamped = Binding<Double>(
    get: { Double(volume) },
    set: { volume = min(10, max(0, Int($0.rounded()))) }
)
Slider(value: clamped, in: 0...10, step: 1)

@Observable vs ObservableObject — the iOS 17 shift

The legacy model: a class: ObservableObject with @Published properties emits objectWillChange for the whole object. Any view observing it via @ObservedObject/@StateObject re-renders when any published property changes, even ones it doesn't read.

The @Observable macro (Observation framework) rewrites stored properties so access is tracked per-property at the point of read. A view that reads only model.count is invalidated only when count changes — not when label does. Under the hood the macro wraps each property with access(keyPath:) / withMutation(keyPath:) calls that register the current observation transaction. This is finer-grained and usually faster, and it removes the @Published boilerplate entirely.

@StateObject vs @ObservedObject — ownership (legacy, still vital)

@StateObject instantiates the object once and keeps it alive across the view's re-renders. @ObservedObject only observes an instance handed in from outside; it does not own its lifetime. The classic bug: declaring @ObservedObject private var vm = ViewModel() means a fresh ViewModel is constructed every time the parent re-renders, silently resetting its state. Rule: the view that creates it uses @StateObject (or @State for @Observable); views that receive it use @ObservedObject (or a plain let).

@Bindable — bindings into an @Observable you don't own

Because an @Observable object is held as a plain property (no wrapper), there's no $ projection by default. @Bindable supplies it: wrap a passed-in observable to get $model.label for a TextField. Use it in child views that edit a model they were given, and inline (@Bindable var x = x) when you need a binding from an environment object.

@EnvironmentObject & @Environment

Both inject dependencies down the tree without prop-drilling. @Environment reads typed values (system ones like \.colorScheme, or custom keys); with @Observable you inject objects via .environment(obj) and read with @Environment(Type.self). Define custom keys with the @Entry macro:

extension EnvironmentValues {
    @Entry var labAccentName: String = "System"
}
// inject:  .environment(\.labAccentName, "Indigo")
// read:    @Environment(\.labAccentName) private var accentName

Performance: minimizing invalidations

Every state change triggers a body recomputation for dependent views and a structural diff. Keep it cheap by: reading the narrowest state possible (@Observable helps automatically); splitting large views so a change touches a small subtree; conforming a view to Equatable + .equatable() so SwiftUI can skip identical re-renders; and using Self._printChanges() in body to see what triggered a redraw (@self, @identity, or a specific property).

Common pitfalls & gotchas

  • @ObservedObject var vm = VM() — creating an owned object in an observing wrapper. Resets on every parent re-render. Use @StateObject/@State.
  • Mutating a copy, not the binding. var local = binding.wrappedValue; local.x = … writes nothing back.
  • Deriving a custom Binding in body that captures stale state — the set closure can clobber newer values. Prefer reading current state inside the closures.
  • Forgetting @ObservationIgnored on stored deps inside an @Observable (e.g. a UserDefaults handle) — harmless but muddies tracking; required for lets you don't want observed.
  • Putting @State on reference types you intend to share. @State is for view-owned sources of truth; shared services belong in the environment.
  • Over-wide views. One giant body reading many properties re-renders wholesale. Decompose.
  • Importing surprise: on Xcode 26 with MEMBER_IMPORT_VISIBILITY, ObservableObject/@Published need an explicit import Combine — SwiftUI no longer re-exports it.

The screen wires five focused demos. Key pieces:

// 1. @State owns; child mutates via @Binding
private struct StateBindingDemo: View {
    @State private var count = 0
    var body: some View {
        VStack(alignment: .leading) {
            Text("Count: \(count)")
            StepperRow(value: $count)   // $count : Binding<Int>
        }
    }
}
private struct StepperRow: View {
    @Binding var value: Int            // observes, does not own
    var body: some View {
        HStack {
            Button("–") { value -= 1 }
            Button("+") { value += 1 }
        }
    }
}

// 3. @Observable owned via @State; child edits via @Bindable
private struct ObservableBindableDemo: View {
    @State private var model = CounterModel()
    var body: some View {
        VStack(alignment: .leading) {
            Text("\(model.label): \(model.count)")
            Button("Increment") { model.increment() }
            LabelEditor(model: model)
        }
    }
}
private struct LabelEditor: View {
    @Bindable var model: CounterModel   // unlocks $model.label
    var body: some View {
        TextField("Label", text: $model.label)
    }
}

// 4. @StateObject OWNS the instance across re-renders
private struct LegacyOwnershipDemo: View {
    @StateObject private var counter = LegacyCounter()
    @State private var tick = 0
    var body: some View {
        VStack {
            Text("Counter: \(counter.count)")
            Button("Increment") { counter.increment() }
            Button("Re-render parent (tick \(tick))") { tick += 1 }
            // Count survives because @StateObject owns it.
            // Swap to @ObservedObject and it resets every tick.
        }
    }
}

// 2. Custom Binding that transforms on write
struct UppercasingField: View {
    @State private var raw = ""
    var body: some View {
        let upper = Binding<String>(
            get: { raw },
            set: { raw = $0.uppercased() }   // transform belongs in the binding
        )
        TextField("Code", text: upper)
        Text("Stored: \(raw)")
    }
}

// 3. @Observable + @Bindable
@Observable final class Account { var displayName = "Ada" ; var isPublic = true }

struct AccountEditor: View {
    @State private var account = Account()           // owner uses @State
    var body: some View {
        Text(account.displayName)                    // tracked read
        NameRow(account: account)                    // pass the instance
    }
}
struct NameRow: View {
    @Bindable var account: Account                   // projects $account.displayName
    var body: some View { TextField("Name", text: $account.displayName) }
}

// 5. Custom environment key
extension EnvironmentValues { @Entry var accountTheme: String = "Default" }
// root:  .environment(\.accountTheme, "Midnight")
// leaf:  @Environment(\.accountTheme) private var theme

Key decisions:

  • The owner of an @Observable uses @State; the editor uses @Bindable on a passed-in instance — never re-instantiate it downstream.
  • The transform lives in the Binding's set, keeping the view body declarative and the storage authoritative.
  • For the legacy ownership task, the only change that matters is @StateObject vs @ObservedObject; the forced re-render (a tick @State) is what exposes the difference.
  • Custom environment keys are for ambient, rarely-changing config (theme, formatting), not for app state you mutate frequently — that belongs in an injected @Observable.
