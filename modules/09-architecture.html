Module 09

Architecture Patterns

Choosing structure deliberately: where MVVM fits and where it's overhead, what TCA buys you, unidirectional data flow with a tiny reducer-backed store, dependency injection without a framework, and SPM feature modules for boundaries at scale.

Concept

MVVM: fit vs overhead

In SwiftUI, the View is already a value-type function of state, and @Observable gives you observable models for free. MVVM fits when a screen has real presentation logic and service calls worth isolating from the view: put that in an @Observable view model. It becomes overhead when the "view model" is a passthrough holding one @State — you've added a class and wiring for nothing. Reach for plain @State/@Observable first; promote to a view model when logic accumulates.

@Observable @MainActor final class SearchViewModel {
    var query = ""
    private(set) var results: [String] = []
    private let corpus: [String]
    func search() { results = corpus.filter { $0.localizedCaseInsensitiveContains(query) } }
}

Unidirectional data flow

State flows down, events (actions) flow up, and a single reducer is the only place state changes. Views send actions; they never mutate state directly. This makes every change traceable and the reducer trivially testable as a pure function. A minimal store:

@Observable final class Store<State, Action> {
    private(set) var state: State
    private let reduce: (inout State, Action) -> Void
    init(initial: State, reduce: @escaping (inout State, Action) -> Void) {
        self.state = initial; self.reduce = reduce
    }
    func send(_ action: Action) { reduce(&state, action) }   // the only mutation path
}

enum CounterFeature {
    struct State: Equatable { var count = 0; var history: [Int] = [] }
    enum Action { case increment, decrement, reset }
    static func reduce(_ s: inout State, _ a: Action) {
        switch a {
        case .increment: s.count += 1; s.history.append(s.count)
        case .decrement: s.count -= 1; s.history.append(s.count)
        case .reset:     s = State()
        }
    }
}

TCA overview

The Composable Architecture formalizes this pattern: State, Action, a Reducer, and an Effect type for side effects, with first-class composition (combining feature reducers), dependency injection, and exhaustive testing of state transitions and effects. The reference store above is the same idea, stripped to its essence. TCA earns its weight on large apps with many shared, interacting features; it's heavy for a handful of screens.

DI without a framework

You don't need a DI container. Two idiomatic patterns: a composition root — an object (here AppContainer) constructed once at launch holding your services, injected via .environment(...) and read with @Environment; and protocol-fronted dependencies with Live/Mock implementations chosen at the edge. Together they give testability and swappability with zero magic.

SPM feature modules

At scale, split features into Swift Package targets with explicit dependencies. The package boundary enforces separation — a feature can't reach into another's internals because it isn't imported. Benefits: faster incremental builds, independent previews, and a forcing function for clean interfaces. Cost: more Package.swift wiring, so modularize along real seams, not speculatively.

Common pitfalls & gotchas

  • View models that do nothing. A class wrapping a single @State adds indirection without isolating logic. Use plain state until there's logic to extract.
  • Mutating state outside the reducer. In a unidirectional store, a view that pokes state directly breaks traceability and testability. Everything goes through send.
  • Adopting TCA for a small app. The boilerplate and learning curve outweigh the benefit until you have many interacting features. Start with the minimal store.
  • Side effects inside the reducer. Keep reducers pure (no I/O, no Date(), no networking) so they're deterministic to test; model effects separately and feed results back as actions.
  • Singletons as hidden dependencies. SomeService.shared sprinkled through views can't be swapped in tests. Inject through the composition root instead.
  • Premature modularization. Splitting into packages before the seams are clear creates churn. Extract a module when a boundary has proven stable.

Challenge

Open Modules/09-Architecture/Challenge/ArchitectureChallengeScreen.swift. Model a todo-toggle feature with the unidirectional Store:

  1. State — { var todos: [String]; var done: Set<Int> }.
  2. Action — add(String), toggle(Int), clearDone.
  3. Reducer — pure reduce(_:_:) implementing each action; expose store().
  4. View — send actions from the UI; render store.state.todos with a per-row toggle. Test the reducer directly.

Solution reveal

Show the solution & commentary 
enum TodoFeature {
    struct State: Equatable { var todos: [String] = []; var done: Set<Int> = [] }
    enum Action { case add(String), toggle(Int), clearDone }

    static func reduce(_ s: inout State, _ a: Action) {
        switch a {
        case .add(let title):
            let t = title.trimmingCharacters(in: .whitespaces)
            guard !t.isEmpty else { return }
            s.todos.append(t)
        case .toggle(let i):
            if s.done.contains(i) { s.done.remove(i) } else { s.done.insert(i) }
        case .clearDone:
            // remove done items high-index first so earlier indices stay valid
            for i in s.done.sorted(by: >) where i < s.todos.count { s.todos.remove(at: i) }
            s.done.removeAll()
        }
    }

    static func store() -> Store<State, Action> { Store(initial: State(), reduce: reduce) }
}

// Reducer test — no store, no UI
@Test func togglesAndClears() {
    var s = TodoFeature.State(todos: ["A", "B"], done: [])
    TodoFeature.reduce(&s, .toggle(0))
    #expect(s.done == [0])
    TodoFeature.reduce(&s, .clearDone)
    #expect(s.todos == ["B"])
    #expect(s.done.isEmpty)
}

Key decisions:

  • The reducer is pure and static — the test calls it with a value and asserts on the result, no async or wiring.
  • clearDone removes by descending index so earlier removals don't shift the indices still to be removed — a classic mutation bug the test guards against.
  • Validation (rejecting empty titles) lives in the reducer, the single source of truth for state changes, not scattered in the view.
