Module 07

SwiftData & Persistence

Modeling with @Model and relationships, fetching with @Query, injecting the ModelContext / ModelContainer, schema migrations, and keeping batch operations and predicates cheap.

Concept

@Model & relationships

@Model is a macro that turns a plain class into a persisted, observable entity — it synthesizes the storage, conformances, and change tracking. Relationships are ordinary properties; annotate them with @Relationship to set the delete rule and the inverse key path, so SwiftData keeps both sides in sync and cascades deletes correctly.

@Model final class TodoProject {
    var name: String
    @Relationship(deleteRule: .cascade, inverse: \TodoItem.project)
    var items: [TodoItem]
    init(name: String, items: [TodoItem] = []) { self.name = name; self.items = items }
}

@Model final class TodoItem {
    var title: String
    var isDone: Bool
    var project: TodoProject?
    init(title: String, isDone: Bool = false) { self.title = title; self.isDone = isDone }
}
Name collision watch: don't call a model Task — it collides with Swift Concurrency's Task. This project uses TodoItem.

@Query fetching

@Query is a view-driven fetch: it reads from the ModelContext in the environment, renders the results, and automatically re-runs and re-renders when the underlying store changes. Sorting and filtering passed to @Query are pushed into the fetch (the store does the work), not applied in Swift after loading everything.

@Query(sort: \TodoItem.title) private var items: [TodoItem]

ModelContext / ModelContainer DI

The container owns the schema and the on-disk store; the context is the scratchpad you read and write through (inserts, deletes, saves). You install a container at some level of the view tree with .modelContainer(...), and descendants read the context via @Environment(\.modelContext). For previews and tests, build an in-memory container so each run is clean and disk-free.

// Self-contained subtree with its own in-memory store
TodoDemoView()
    .modelContainer(for: [TodoProject.self, TodoItem.self], inMemory: true)

// Writing through the context
@Environment(\.modelContext) private var context
context.insert(TodoItem(title: "Buy milk"))
context.delete(item)
// mutating a fetched @Model autosaves on the next runloop tick

Migrations & schema versions

Lightweight changes (adding an optional property, a new model) migrate automatically. Anything that changes meaning — renames, type changes, splitting an entity — needs an explicit VersionedSchema per version plus a SchemaMigrationPlan describing lightweight or custom stages between them. Version your schema from the first release so you have a migration anchor.

Batch ops & predicate cost

A #Predicate compiles to a store query (SQL under the hood), so the filter runs in the database rather than in Swift — keep filtering in the FetchDescriptor, not in a post-fetch .filter { } that loads everything first. For large deletes, context.delete(model:where:) performs a batch delete without materializing every object.

FetchDescriptor<TodoItem>(
    predicate: #Predicate { $0.isDone == false },   // runs in the store
    sortBy: [SortDescriptor(\.title)]
)

Common pitfalls & gotchas

  • Naming a model Task. Collides with concurrency's Task and produces baffling errors. Pick a distinct name.
  • Filtering after the fetch. @Query all rows then .filter { } in Swift loads the whole table. Push the predicate into the query so the store filters.
  • Missing inverse relationship. Without inverse:, the two sides can drift and cascade deletes may not fire. Always declare the inverse.
  • Forgetting import SwiftData. .modelContainer and @Query are unavailable without it — even in a #Preview.
  • Sharing one context across threads. A ModelContext is not thread-safe. Use the main context for UI; create a separate context (or a ModelActor) for background work.
  • Assuming every schema change is automatic. Renames and type changes need an explicit migration plan or the store fails to open. Version early.

// Centralized container/context factories — app, previews, and tests build the schema once
enum TodoStore {
    static func inMemoryContainer() throws -> ModelContainer {
        try ModelContainer(for: TodoProject.self, TodoItem.self,
                           configurations: ModelConfiguration(isStoredInMemoryOnly: true))
    }
    static func openItemsDescriptor() -> FetchDescriptor<TodoItem> {
        FetchDescriptor(predicate: #Predicate { $0.isDone == false },
                        sortBy: [SortDescriptor(\.title)])
    }
}

Solution reveal

Show the solution & commentary 
@Model final class Note {
    var body: String
    var pinned: Bool
    var createdAt: Date
    init(body: String, pinned: Bool = false, createdAt: Date = .now) {
        self.body = body; self.pinned = pinned; self.createdAt = createdAt
    }
}

struct SwiftDataChallengeScreen: View {
    var body: some View {
        NotebookView().modelContainer(for: [Note.self], inMemory: true)
    }
}

private struct NotebookView: View {
    @Environment(\.modelContext) private var context
    @Query(sort: \Note.createdAt, order: .reverse) private var notes: [Note]
    @State private var draft = ""
    @State private var pinnedOnly = false

    private var visible: [Note] { pinnedOnly ? notes.filter(\.pinned) : notes }

    var body: some View {
        List {
            Section {
                HStack {
                    TextField("New note", text: $draft).textFieldStyle(.roundedBorder)
                    Button("Add") {
                        context.insert(Note(body: draft)); draft = ""
                    }.disabled(draft.isEmpty)
                }
                Toggle("Pinned only", isOn: $pinnedOnly)
            }
            ForEach(visible) { note in
                Button { note.pinned.toggle() } label: {
                    Label(note.body, systemImage: note.pinned ? "pin.fill" : "pin")
                }
            }
            .onDelete { for i in $0 { context.delete(visible[i]) } }
        }
    }
}

Key decisions:

  • @Query(order: .reverse) sorts in the store; no manual sort after fetch.
  • Toggling note.pinned mutates a live @Model — SwiftData tracks the change and autosaves; the @Query re-renders.
  • For a large notebook, replace the in-Swift visible filter with a @Query built from a #Predicate { $0.pinned } so the database does the filtering.
  • An in-memory container keeps the demo disposable and never touches the app's real store.
