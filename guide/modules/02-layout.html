Module 02

Layout & Rendering

How SwiftUI measures and places views — from the Layout protocol to PreferenceKey, alignment guides, view identity, and the @ViewBuilder / AnyView trade-off.

Concept

The Layout protocol vs GeometryReader

SwiftUI's layout pass is a two-step negotiation: parent proposes a size, child reports what it needs, parent places it. The Layout protocol (iOS 16+) lets you write a custom container that participates in this negotiation cleanly — sizeThatFits answers "how big do you need to be?", placeSubviews positions each child within the allocated bounds.

GeometryReader is a frequent misuse for layout tasks. It breaks the top-down flow: it reads its proposed size (not the natural size of its children), meaning you often need a two-pass layout — measure children, then re-render with a hard-coded size. It also leaks coordinate spaces into the view tree. Use GeometryReader for reading geometry after layout (e.g. positioning an overlay relative to a frame). Use the Layout protocol for computing layout.

struct EqualWidthVStack: Layout {
    var spacing: CGFloat = 8

    func sizeThatFits(
        proposal: ProposedViewSize,
        subviews: Subviews,
        cache: inout ()
    ) -> CGSize {
        guard !subviews.isEmpty else { return .zero }
        let w = subviews.map { $0.sizeThatFits(.unspecified).width }.max()!
        let heights = subviews.map { $0.sizeThatFits(ProposedViewSize(width: w, height: nil)).height }
        let totalH = heights.reduce(0, +) + spacing * CGFloat(subviews.count - 1)
        return CGSize(width: proposal.width ?? w, height: totalH)
    }

    func placeSubviews(
        in bounds: CGRect,
        proposal: ProposedViewSize,
        subviews: Subviews,
        cache: inout ()
    ) {
        let w = subviews.map { $0.sizeThatFits(.unspecified).width }.max()!
        var y = bounds.minY
        for subview in subviews {
            let h = subview.sizeThatFits(ProposedViewSize(width: w, height: nil)).height
            subview.place(
                at: CGPoint(x: bounds.midX, y: y + h / 2),
                anchor: .center,
                proposal: ProposedViewSize(width: w, height: h)
            )
            y += h + spacing
        }
    }
}

PreferenceKey — child-to-parent communication

SwiftUI's layout data flows top-down (parent → child). PreferenceKey is the sanctioned bottom-up channel: a child writes a value via .preference(key:value:) (usually from inside a GeometryReader background), the framework reduces all values in the subtree using reduce, and the nearest ancestor that calls .onPreferenceChange receives the result.

The canonical use: making a column of labels all share the width of the widest one, without a shared @State binding. Each label reports its natural width; the parent reads the max and feeds it back as frame(minWidth:).

struct MaxWidthKey: PreferenceKey {
    static let defaultValue: CGFloat = 0
    static func reduce(value: inout CGFloat, nextValue: () -> CGFloat) {
        value = max(value, nextValue())
    }
}

// In the child:
Text(key)
    .fixedSize()
    .background(GeometryReader { geo in
        Color.clear.preference(key: MaxWidthKey.self, value: geo.size.width)
    })

// In the parent:
.onPreferenceChange(MaxWidthKey.self) { maxWidth = $0 }

alignmentGuide & custom alignments

Every built-in alignment guide (.top, .center, .firstTextBaseline, …) has a default value for each view. .alignmentGuide overrides that value for a single view — the closure receives the view's ViewDimensions and returns the new offset along the guide's axis.

Custom alignments let you define a new shared axis that multiple views can opt into. Define one with an AlignmentID, then use it in an HStack(alignment:) or VStack(alignment:):

extension VerticalAlignment {
    private enum IconMidID: AlignmentID {
        static func defaultValue(in context: ViewDimensions) -> CGFloat {
            context[VerticalAlignment.center]  // fallback
        }
    }
    static let iconMid = VerticalAlignment(IconMidID.self)
}

HStack(alignment: .iconMid) {
    Image(systemName: "bell.badge.fill")
        .alignmentGuide(.iconMid) { d in d[VerticalAlignment.center] }
    VStack(alignment: .leading) {
        Text("Title").font(.headline)
        Text("Subtitle").font(.caption)
            .alignmentGuide(.iconMid) { d in d[VerticalAlignment.center] }
    }
}

Structural vs explicit identity

SwiftUI assigns every view an identity that governs whether it receives a fresh instance or has its state preserved. Structural identity is inferred from the view's position in the body tree — a view at the same position across renders is the same view, and its @State survives.

Explicit identity is set with .id(_:). When the value changes, SwiftUI treats the view as a different entity, destroys the old one, and inserts a fresh one — resetting all @State. This is useful for "reset this form" behavior.

if/else ≠ same structural position. if condition { ViewA() } else { ViewA() } puts ViewA at two different structural positions (the true-branch and the false-branch). Toggling the condition does discard and recreate the view. If you need to preserve state across a condition, change a prop instead of branching: ViewA(param: condition ? x : y).

@ViewBuilder & AnyView trade-offs

@ViewBuilder is a result-builder that turns a block of conditional view expressions into a statically-typed composite. The exact type is visible to the SwiftUI differ at compile time, enabling structural diffing and animation between branches.

AnyView type-erases its contents. SwiftUI sees an opaque box — it can't look inside to diff the subtrees, so it replaces the entire box on every change. Animations across AnyView boundaries don't interpolate; performance suffers in tight loops.

@ViewBuilderAnyView
Type visible to SwiftUIYes — exact composite typeNo — opaque box
Structural diffingYesNo
Transitions across branchesInterpolatedWhole-box replace
When to useAll conditional view compositionHeterogeneous collections, unavoidable type erasure

Common pitfalls & gotchas

  • GeometryReader for sibling sizing. GeometryReader reports the proposed size, not the natural size of its children. Reading sibling dimensions from it in layout code almost always needs a two-pass re-render. Use the Layout protocol instead.
  • PreferenceKey feedback loops. If setting the collected preference value triggers a re-render that changes the reported value (e.g. frame(width: collectedWidth) changes the child's reported width), you get an oscillation. Report natural sizes (from a fixedSize or hidden overlay), not the constrained frame size.
  • if/else creates two structural positions. Each branch is a distinct position — state doesn't survive the branch flip. Use prop-based variation (View(flag: condition)) to preserve state across condition changes.
  • AnyView inside ForEach / List. List diffs its rows structurally. Wrapping rows in AnyView forces full replacement on every change instead of in-place animation.
  • Animating across AnyView boundaries. A transition on an AnyView doesn't interpolate between the old and new content — it replaces the box. Unwrap to a @ViewBuilder function to get real interpolation.
  • Layout sizeThatFits proposals. The system may call sizeThatFits with .unspecified (asking for the natural size), .zero (asking for the minimum), or a concrete size. Treat proposal.width == nil as "use natural width" and avoid crashing on edge cases.

// 1. EqualHeightHStack
struct EqualHeightHStack: Layout {
    var spacing: CGFloat = 8

    func sizeThatFits(proposal: ProposedViewSize, subviews: Subviews, cache: inout ()) -> CGSize {
        guard !subviews.isEmpty else { return .zero }
        let maxH = subviews.map { $0.sizeThatFits(.unspecified).height }.max()!
        let totalW = subviews.map { $0.sizeThatFits(ProposedViewSize(width: nil, height: maxH)).width }.reduce(0, +)
                     + spacing * CGFloat(subviews.count - 1)
        return CGSize(width: proposal.width ?? totalW, height: maxH)
    }

    func placeSubviews(in bounds: CGRect, proposal: ProposedViewSize, subviews: Subviews, cache: inout ()) {
        let maxH = subviews.map { $0.sizeThatFits(.unspecified).height }.max()!
        var x = bounds.minX
        for subview in subviews {
            let w = subview.sizeThatFits(ProposedViewSize(width: nil, height: maxH)).width
            subview.place(
                at: CGPoint(x: x + w / 2, y: bounds.midY),
                anchor: .center,
                proposal: ProposedViewSize(width: w, height: maxH)
            )
            x += w + spacing
        }
    }
}

struct MaxHeightKey: PreferenceKey {

// 3. Custom HorizontalAlignment
extension HorizontalAlignment {
    private enum TrailingLabelID: AlignmentID {
        static func defaultValue(in context: ViewDimensions) -> CGFloat { context[HorizontalAlignment.trailing] }
    }
    static let trailingLabel = HorizontalAlignment(TrailingLabelID.self)
}
// Usage:
VStack(alignment: .trailingLabel) {
    HStack { Image(systemName: "wifi"); Text("Wi-Fi").alignmentGuide(.trailingLabel) { d in d[.trailing] } }
    Text("Bluetooth").alignmentGuide(.trailingLabel) { d in d[.trailing] }
}

// 4. Explicit identity
TextField("Type something…", text: $note)
    .textFieldStyle(.roundedBorder)
    .id(entryKey)   // ← this is the only change

// 5. SignalStrength with @ViewBuilder
struct SignalStrength: View {
    let bars: Int
    var body: some View {
        icon.font(.title2).foregroundStyle(bars == 0 ? .secondary : .primary)
    }

    @ViewBuilder private var icon: some View {
        switch bars {
        case 0:  Image(systemName: "wifi.slash")
        case 1:  Image(systemName: "wifi.exclamationmark")
        case 2:  Image(systemName: "wifi")
        default: Image(systemName: "wifi").symbolRenderingMode(.hierarchical)
        }
    }
}

Key decisions:

  • In EqualHeightHStack, compute maxH from .unspecified proposals (natural size), then re-query each child's width at that height — some views (e.g. text) change width when height changes.
  • The .trailingLabel alignment works because AlignmentID.defaultValue fires for any view that doesn't explicitly opt in — you only need to call .alignmentGuide on the ones that differ from the default.
  • .id(entryKey) on the TextField is the entire fix — the $note binding's value is irrelevant once the field is replaced with a new instance.
  • @ViewBuilder private var icon keeps the type visible and the switch exhaustive. A switch inside @ViewBuilder is fully supported and the cleanest pattern for finite enumerations of views.
