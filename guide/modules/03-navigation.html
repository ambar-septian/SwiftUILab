Module 03

Navigation

Type-safe, data-driven navigation with NavigationStack and NavigationPath — a single source of truth for the stack, programmatic deep links, the coordinator pattern, and enum-driven modals.

Concept

NavigationStack + NavigationPath

The modern navigation model (iOS 16+) is state-driven: you bind a NavigationStack to an array (or a type-erased NavigationPath) that represents the pushed screens. Appending pushes; removing pops. The view is a pure function of that array — there is no imperative pushViewController. This is the inverse of UIKit's UINavigationController, where the stack is owned internally and you mutate it with method calls.

@Observable final class ShopCoordinator {
    var path: [ShopRoute] = []
}

NavigationStack(path: $coordinator.path) {
    RootView()
        .navigationDestination(for: ShopRoute.self) { route in
            // exhaustive switch → every route resolves to a screen
        }
}

Use a typed array ([ShopRoute]) when every destination is the same type — you get Codable state restoration and easy inspection. Use NavigationPath when the stack mixes heterogeneous value types.

Type-safe routes

Model every push destination as one Hashable enum. Pushing is path.append(.product(p)) or a value-based NavigationLink(value:); the navigationDestination(for:) switch maps each case to a screen. Because the enum is exhaustive, the compiler guarantees you can't forget a destination — the class of "dead link" bug that plagues string-based routing simply can't compile.

enum ShopRoute: Hashable {
    case category(String)
    case product(Product)
}

NavigationLink(value: ShopRoute.category("Phones")) { Text("Phones") }

Programmatic deep linking

Because the stack is data, a deep link is just an assignment. Building the whole array at once synthesizes the back-stack, so the system back button walks the hierarchy as if the user had tapped in manually.

func deepLink(to product: Product) {
    path = [.category(product.category), .product(product)]  // two screens deep
}

Keep the URL→routes mapping in a pure function, free of SwiftUI, so it's unit-testable in isolation:

enum DeepLink {
    static func routes(from url: URL) -> [ShopRoute] {
        guard url.scheme == "swiftuilab", let host = url.host else { return [] }
        // … map host + path components to routes
    }
}

The coordinator pattern

Putting path (and any modal state) on one @Observable object placed in the environment lets any descendant drive navigation without threading bindings down through every initializer. Child screens read @Environment(ShopCoordinator.self) and call coordinator.openProduct(p). This centralizes flow logic, keeps leaf views dumb, and makes the navigation testable as plain method calls on the coordinator.

Sheet / fullScreenCover / popover state

Modal presentation is also state-driven. Prefer sheet(item:) over sheet(isPresented:) when the modal needs data: bind an optional Identifiable enum — assigning presents, setting nil dismisses. One source of truth, no separate boolean to drift out of sync with "which thing is showing."

enum ShopSheet: String, Identifiable { case cart, about; var id: String { rawValue } }

.sheet(item: $coordinator.sheet) { sheet in SheetContent(sheet: sheet) }

Common pitfalls & gotchas

  • NavigationView is deprecated. It had ambiguous column behavior and no programmatic stack. Use NavigationStack (single column) or NavigationSplitView (multi-column) instead.
  • navigationDestination inside a lazy container. Attaching .navigationDestination inside a List row or LazyVStack that may not be rendered means the modifier isn't registered and the link silently does nothing. Attach it once, high in the stack's content.
  • Multiple navigationDestination for the same type. Two registrations for the same value type in one stack is undefined — the last one wins unpredictably. Keep one switchboard per type.
  • Non-Hashable route values. Everything you push must be Hashable; a stored associated value that isn't (e.g. a closure) won't compile or will break diffing.
  • Boolean modal soup. A separate @State var showCart, showAbout, … invites "two sheets at once" bugs. Collapse them into one optional enum with sheet(item:).
  • Resetting the path mid-animation. Mutating path while a push/pop is animating can drop frames or desync. Drive it from one place (the coordinator) and avoid clobbering it from multiple callbacks.

// Coordinator owns the stack + modal state, lives in the environment
@Observable final class ShopCoordinator {
    var path: [ShopRoute] = []
    var sheet: ShopSheet?
    func deepLink(to product: Product) { path = [.category(product.category), .product(product)] }
    func popToRoot() { path.removeAll() }
}

// The stack: typed path + exhaustive destination switch + enum-driven sheet
NavigationStack(path: $coordinator.path) {
    List { /* category links, deep-link buttons, sheet buttons */ }
        .navigationDestination(for: ShopRoute.self) { route in
            switch route {
            case .category(let name): CategoryScreen(category: name)
            case .product(let p):     ProductScreen(product: p)
            }
        }
        .sheet(item: $coordinator.sheet) { SheetContent(sheet: $0) }
}
.environment(coordinator)

Challenge

Open Modules/03-Navigation/Challenge/NavigationChallengeScreen.swift. Build a type-safe Settings flow:

  1. Route + sheet enums — model every push as SettingsRoute: Hashable and the modal as SettingsSheet: Identifiable.
  2. Coordinator — an @Observable object holding path and sheet with push / popToRoot / deepLink helpers.
  3. Stack wiring — bind NavigationStack(path:), register one exhaustive navigationDestination, present with sheet(item:).
  4. Deep link — a pure SettingsDeepLink.routes(from:) mapping swiftuilab://settings/<section> to the route stack.

Constraint: keep the file compiling at every step.

Solution reveal

Show the solution & commentary 
enum SettingsRoute: Hashable {
    case section(String)
    case detail(String)
}

enum SettingsSheet: String, Identifiable, CaseIterable {
    case about
    var id: String { rawValue }
}

@Observable final class SettingsCoordinator {
    var path: [SettingsRoute] = []
    var sheet: SettingsSheet?
    func open(_ section: String) { path.append(.section(section)) }
    func deepLink(to section: String) { path = [.section(section), .detail(section)] }
    func popToRoot() { path.removeAll() }
}

enum SettingsDeepLink {
    static func routes(from url: URL) -> [SettingsRoute] {
        guard url.scheme == "swiftuilab", url.host == "settings" else { return [] }
        guard let section = url.pathComponents.filter({ $0 != "/" }).first else { return [] }
        return [.section(section), .detail(section)]
    }
}

// View
struct SettingsFlow: View {
    @State private var coordinator = SettingsCoordinator()
    var body: some View {
        @Bindable var coordinator = coordinator
        NavigationStack(path: $coordinator.path) {
            List {
                ForEach(["Account", "Notifications", "Privacy"], id: \.self) { s in
                    NavigationLink(value: SettingsRoute.section(s)) { Text(s) }
                }
                Button("Deep link to Privacy") { coordinator.deepLink(to: "Privacy") }
                Button("About") { coordinator.sheet = .about }
            }
            .navigationDestination(for: SettingsRoute.self) { route in
                switch route {
                case .section(let s): Text("\(s) settings")
                case .detail(let s):  Text("\(s) detail")
                }
            }
            .sheet(item: $coordinator.sheet) { _ in Text("About SwiftUILab") }
        }
        .environment(coordinator)
    }
}

Key decisions:

  • The route enum is the contract: the navigationDestination switch is exhaustive, so adding a case forces you to handle it.
  • deepLink assigns the whole array so the synthesized back stack is correct — the user can pop back through section after a direct jump to detail.
  • The parser is pure and returns [] for anything it can't resolve, so a malformed URL is a no-op rather than a crash — and it's testable without a view.
  • @Bindable var coordinator inside body is what unlocks $coordinator.path for the stack binding from an @Observable object.
