Module 10

Performance & Debugging

Finding and fixing the things that make a SwiftUI app feel slow: redraw debugging with Self._printChanges() and the SwiftUI Instruments template, scoping invalidations, hunting Task and closure leaks, detecting hitches and hangs, and trimming launch time.

Concept

The SwiftUI Instruments template

The SwiftUI template in Instruments (with View Body, View Properties, and Core Animation Commits lanes) shows how often each view's body runs, why it re-ran (which dependency changed), and how long commits take. Start here before optimizing — measure which bodies are hot rather than guessing. Pair it with Time Profiler for CPU and Animation Hitches for dropped frames.

Self._printChanges()

A debugging call you drop at the top of a body: it logs, on every invocation, the dependency that forced the re-run (@self for identity changes, the property name for state changes). It's the fastest way to answer "why is this view re-rendering?" — but it's a private API, so remove it before shipping.

var body: some View {
    let _ = Self._printChanges()        // logs the trigger each render; debug only
    return VStack { /* … */ }
}

Scoping invalidations

SwiftUI re-runs a body when a dependency it read changes. The lever is granularity: read state as low in the tree as possible. Extracting a frequently-changing piece into its own small subview means only that subview re-runs, not its parent and all siblings. With @Observable, observation is per-property — a view that reads only settings.username doesn't re-render when settings.notificationsEnabled changes (see Module 1's invalidation demo).

// Bad: parent reads count, so the whole section re-renders on every tick
Text("count = \(count)")

// Better: isolate the volatile read in a leaf
struct CounterLabel: View { let count: Int; var body: some View { Text("count = \(count)") } }

Task & closure leaks

The most common leak is a retain cycle: an object stores a closure that captures self strongly, so the object keeps itself alive forever. The fix is [weak self] at the capture site. The second is an unstructured Task started from a view that outlives it — prefer .task (auto-cancelled) over Task { } (you must cancel). Verify with the Leaks and Allocations instruments, or in a unit test via a weak reference after dropping the strong one.

final class RetainNode {
    var onChange: (() -> Void)?
    func leakySetup() { onChange = { self.value += 1 } }              // ❌ self → onChange → self
    func safeSetup()  { onChange = { [weak self] in self?.value += 1 } } // ✅ no cycle
}

Hitches & hang detection

A hitch is a frame that misses its budget (~16.67ms at 60fps, ~8.3ms at 120fps), seen as a stutter. A hang is the main thread blocked long enough that the app stops responding. Causes: heavy work in body or on the main actor, synchronous I/O, oversized images. Detect hitches with the Animation Hitches instrument and Xcode's Hangs organizer; fix by moving work off the main actor and keeping body cheap and pure.

Launch time

Launch is split into pre-main (dynamic linking, static initializers) and post-main (your first frame). Keep work out of init/App.init, defer non-critical setup until after first paint, lazy-load heavy services, and avoid synchronous disk/network on the launch path. Measure with the App Launch instrument and the MetricKit/Xcode launch-time metrics.

Common pitfalls & gotchas

  • Expensive work in body. body can run many times per second; sorting, formatting, or decoding there tanks performance. Precompute and cache outside body.
  • Over-wide observation. A view reading a whole model re-renders on any change. Read the narrowest slice, and split volatile reads into leaf subviews.
  • AnyView in hot paths. Type erasure disables structural diffing, forcing full subtree replacement (see Module 2). Avoid it in lists and frequently-updated views.
  • Strong self in stored closures. The retain-cycle leak. Use [weak self] and verify deallocation.
  • Task { } instead of .task. Unstructured tasks from a view leak work past the view's lifetime. Prefer .task for auto-cancellation.
  • Shipping _printChanges / debug logging. It's private API and noise in production. Strip it.
  • Optimizing without measuring. Guessing wastes effort and can pessimize. Profile in Instruments first, then fix the proven hotspot.

// Pure, testable hitch detector — rolling average vs the frame budget
struct FrameMonitor {
    let budgetMs = 1000.0 / 60.0
    private(set) var samples: [Double] = []
    mutating func record(_ ms: Double) { samples.append(ms); /* trim to window */ }
    var average: Double { samples.isEmpty ? 0 : samples.reduce(0, +) / Double(samples.count) }
    var isHitchy: Bool { average > budgetMs }
}

// Live retain-cycle demo: leaky vs safe, checked with a weak reference
func run(safe: Bool) -> String {
    var node: RetainNode? = RetainNode()
    weak var weakNode = node
    if safe { node?.safeSetup() } else { node?.leakySetup() }
    node = nil
    return weakNode == nil ? "Deallocated ✅" : "Leaked ❌"
}

Solution reveal

Show the solution & commentary 
// 1 + 2. Scope the redraw and observe it
private struct CounterLabel: View {
    let count: Int
    var body: some View {
        let _ = Self._printChanges()        // only logs when `count` actually changes
        Text("count = \(count)").font(.title3.monospacedDigit())
    }
}

struct Section: View {
    @State private var count = 0
    var body: some View {
        let _ = Self._printChanges()        // parent: logs on its own state changes
        return VStack {
            CounterLabel(count: count)      // isolated leaf re-renders independently
            Button("Increment") { count += 1 }
        }
    }
}

// 3. Fix the cycle
final class BrokenTimer {
    var tick: (() -> Void)?
    private(set) var ticks = 0
    func fixed() { tick = { [weak self] in self?.ticks += 1 } }   // breaks the cycle
}

@Test func deallocates() {
    var t: BrokenTimer? = BrokenTimer()
    weak var weakT = t
    t?.fixed()
    t = nil
    #expect(weakT == nil)               // safe setup → node is gone
}

Key decisions:

  • Moving the volatile count read into CounterLabel means the parent's body doesn't depend on it — only the leaf re-runs.
  • Two _printChanges (parent + child) make the scoping visible: you see exactly which body fires on a tap.
  • The weak-reference test is a deterministic, Instruments-free way to assert no retain cycle — drop the strong reference and require the weak one to be nil.
