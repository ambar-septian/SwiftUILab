Module 06

UIKit Interop

Bridging both directions: wrapping UIKit views and controllers for SwiftUI with UIViewRepresentable / UIViewControllerRepresentable and the Coordinator, avoiding update loops, hosting SwiftUI in UIKit with UIHostingController, and incremental migration.

Concept

UIViewRepresentable + Coordinator

A UIViewRepresentable adapts a single UIKit view into the SwiftUI tree via three methods: makeUIView (create it once), updateUIView (push the latest SwiftUI state in, on every render), and makeCoordinator (create the bridge object). The Coordinator is an NSObject that owns the imperative glue: it holds the SwiftUI @Binding and serves as the target for target-action, or the delegate for UIKit callbacks, translating UIKit events back into SwiftUI state.

struct RatingSlider: UIViewRepresentable {
    @Binding var value: Double

    func makeCoordinator() -> Coordinator { Coordinator(value: $value) }

    func makeUIView(context: Context) -> UISlider {
        let slider = UISlider()
        slider.addTarget(context.coordinator,
                         action: #selector(Coordinator.valueChanged(_:)),
                         for: .valueChanged)
        return slider
    }

    func updateUIView(_ uiView: UISlider, context: Context) { /* push value in */ }

    final class Coordinator: NSObject {
        let value: Binding<Double>
        init(value: Binding<Double>) { self.value = value }
        @objc func valueChanged(_ sender: UISlider) { value.wrappedValue = Double(sender.value) }
    }
}

Avoiding infinite update loops

The danger zone is the two-way path. updateUIView runs on every SwiftUI render and pushes state into the UIKit view; the UIKit view's callback pushes back into SwiftUI state, which triggers another render, which calls updateUIView again. If you write the UIKit value unconditionally — especially mid-gesture — you can fight the user or oscillate. The fix is to guard the write: only apply the incoming value when it actually differs from the view's current value.

func updateUIView(_ uiView: UISlider, context: Context) {
    guard abs(Double(uiView.value) - value) > 0.0001 else { return }  // loop breaker
    uiView.setValue(Float(value), animated: false)
}

UIViewControllerRepresentable lifecycle

Same shape, one level up: makeUIViewController builds the controller; updateUIViewController pushes SwiftUI state in on each render. The controller's own lifecycle (viewDidLoad, viewWillAppear, …) runs normally. Use this for whole UIKit screens you can't yet rewrite — camera pickers, web views, complex custom controllers.

UIHostingController sizing

Going the other way, UIHostingController embeds a SwiftUI view inside UIKit. The classic bug is sizing: in a UITableViewCell or stack view, the host may report a zero/too-small height. Enable intrinsic sizing — set hostingController.sizingOptions = [.intrinsicContentSize] (iOS 16+) so Auto Layout reads SwiftUI's ideal size, or call invalidateIntrinsicContentSize() when content changes.

Migration strategies

Interop enables incremental adoption rather than a rewrite: host new SwiftUI screens inside an existing UIKit navigation flow with UIHostingController, and wrap legacy UIKit views you still need inside new SwiftUI screens with representables. Migrate leaf-first (small reusable views) before tackling whole flows.

Common pitfalls & gotchas

  • Unconditional writes in updateUIView. The infinite-loop / gesture-fighting bug. Guard the write so you only apply genuinely-changed values.
  • State on the representable struct. The struct is recreated every render; put persistent state and references on the Coordinator (or in SwiftUI @State), never as stored vars on the representable.
  • Retain cycles via the Coordinator. A Coordinator strongly held by the view while also holding a closure back to it can leak. Keep the binding (a value) and use weak for delegate back-references where needed.
  • Heavy work in makeUIView. It runs once, but expensive setup there blocks the first render. Defer non-essential work.
  • Zero-height UIHostingController. Forgetting sizingOptions = [.intrinsicContentSize] in cells yields collapsed content. Enable intrinsic sizing.
  • Coordinator not used as the delegate target. Setting a delegate to self (the struct) instead of context.coordinator means callbacks never fire — the struct is gone by then.

enum UpdateGuard {

    }

struct WrappedTextField: UIViewRepresentable {
    @Binding var text: String

    func makeCoordinator() -> Coordinator { Coordinator(text: $text) }

    func makeUIView(context: Context) -> UITextField {
        let field = UITextField()
        field.borderStyle = .roundedRect
        field.addTarget(context.coordinator,
                        action: #selector(Coordinator.editingChanged(_:)),
                        for: .editingChanged)
        return field
    }

    func updateUIView(_ uiView: UITextField, context: Context) {
        if uiView.text != text { uiView.text = text }   // guard: don't clobber while typing
    }

    final class Coordinator: NSObject {
        let text: Binding<String>
        init(text: Binding<String>) { self.text = text }
        @objc func editingChanged(_ sender: UITextField) { text.wrappedValue = sender.text ?? "" }
    }
}

final class PageViewController: UIViewController {
    var page = 0 { didSet { label.text = "Page \(page)" } }
    private let label = UILabel()
    override func viewDidLoad() {
        super.viewDidLoad()
        label.text = "Page \(page)"; label.font = .boldSystemFont(ofSize: 28)
        label.translatesAutoresizingMaskIntoConstraints = false
        view.addSubview(label)
        NSLayoutConstraint.activate([
            label.centerXAnchor.constraint(equalTo: view.centerXAnchor),
            label.centerYAnchor.constraint(equalTo: view.centerYAnchor),
        ])
    }
}

struct PageControllerView: UIViewControllerRepresentable {
    @Binding var page: Int
    func makeUIViewController(context: Context) -> PageViewController { PageViewController() }
    func updateUIViewController(_ vc: PageViewController, context: Context) { vc.page = page }
}

Key decisions:

  • The updateUIView guard (uiView.text != text) is the whole game — without it, a re-render mid-edit resets the cursor or wipes input.
  • The Coordinator is the target, created once via makeCoordinator; the representable struct is transient and can't be a stable target.
  • The controller's didSet keeps its label in sync from one place, so updateUIViewController only needs to assign page.
