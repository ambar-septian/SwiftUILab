Module 05

Structured Concurrency & Actors

Fanning work out with TaskGroup and async let, protecting mutable state with actors, understanding reentrancy and @MainActor isolation, custom global actors, Sendable, and what Swift 6 strict concurrency actually enforces.

Concept

TaskGroup vs async let

Both run child tasks concurrently under structured concurrency — no child outlives the scope that created it, and cancelling the parent cancels every child. The difference is shape:

  • async let — a fixed, small number of parallel jobs known at compile time. Bind them, then await the results. Reads like ordinary code.
  • TaskGroup — a dynamic collection of homogeneous jobs (a loop over a list). Add tasks, then reduce results as they complete (in nondeterministic order).
// async let — three concurrent loads, joined
async let users = slowCount(42)
async let revenue = slowCount(1000)
async let errors = slowCount(3)
return await DashboardSnapshot(users: users, revenue: revenue, errors: errors)

// TaskGroup — N concurrent jobs, reduced
await withTaskGroup(of: Int.self) { group in
    for v in values { group.addTask { v * v } }
    var total = 0
    for await partial in group { total += partial }
    return total
}

Actors & data-race safety

An actor serializes access to its mutable state: concurrent callers queue, and only one runs inside the actor at a time. This replaces locks/serial queues around shared state — the compiler enforces that you can only touch an actor's mutable state through await, so data races on that state are impossible by construction.

Actor reentrancy

Actors are reentrant: when an actor method hits an await and suspends, the actor is free to process other messages before the first method resumes. So any invariant you checked before an await may be false after it. Always re-validate state across a suspension point inside an actor.

func withdrawSlowly(_ amount: Int) async -> Bool {
    guard balance >= amount else { return false }
    await Task.yield()                      // other messages may run here
    guard balance >= amount else { return false }  // re-check — state may have changed
    balance -= amount
    return true
}

@MainActor isolation

@MainActor is a global actor whose executor is the main thread. Anything annotated with it (or, in this project, anything left unannotated — the build sets SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor) runs on the main thread, so UI updates are safe by default. Crossing from a background context into @MainActor requires an await, which the compiler inserts and checks.

Global actors

You can declare your own global actor with @globalActor — a single shared serial executor that any number of types can opt into, the same way @MainActor works for the UI. Useful for serializing a subsystem (a database layer, an audit log) without passing one actor instance around.

@globalActor actor DataActor { static let shared = DataActor() }

@DataActor final class AuditLog { /* all access runs on DataActor's executor */ }

Sendable & Swift 6 strict concurrency

Sendable marks a type safe to pass across isolation domains. Value types of Sendable members are automatically Sendable; reference types must be immutable, internally synchronized, or @unchecked Sendable (you promise correctness). Swift 6's strict concurrency turns data-race risks into compile errors: passing a non-Sendable mutable class between actors won't build. This project runs Swift 5 + approachable concurrency; treat this module as the place to write code that would also pass under Swift 6.

Common pitfalls & gotchas

  • Assuming actors are non-reentrant. The "check-then-act" you wrote before an await can be invalidated by an interleaved call. Re-check after every suspension.
  • Actor hopping in a hot loop. Each await someActor.x is a potential suspension and context switch. Batch work into one actor method instead of many fine-grained awaits.
  • @unchecked Sendable as a silencer. It suppresses the compiler's check but not the underlying race. Only use it when you've actually made the type thread-safe (e.g. a lock inside).
  • Blocking inside an actor. A synchronous blocking call (sleep(), a semaphore wait) holds the actor's executor and stalls every queued message. Use await Task.sleep and async APIs.
  • Capturing main-actor state in a detached task. Task.detached drops the actor context; touching @MainActor state inside it requires an explicit hop and may race if you force it.
  • Unstructured tasks for parallel work. Spawning loose Task { }s loses cancellation propagation and structured guarantees. Prefer async let / TaskGroup.

Solution reveal

Show the solution & commentary 
// 1. TaskGroup max
func parallelMax(_ values: [Int]) async -> Int? {
    await withTaskGroup(of: Int.self) { group in
        for v in values { group.addTask { v } }
        var best: Int? = nil
        for await v in group { best = max(best ?? v, v) }
        return best
    }
}

// 2. async let join
func loadProfile() async -> (name: String, posts: Int) {
    async let name = slowName()
    async let posts = slowPosts()
    return await (name, posts)
}

// 3. Actor counter
actor Counter {
    private(set) var value = 0
    func increment() { value += 1 }
}
let counter = Counter()
await withTaskGroup(of: Void.self) { group in
    for _ in 0..<500 { group.addTask { await counter.increment() } }
}
total = await counter.value   // exactly 500

Key decisions:

  • The group's reduction must be commutative/associative because children complete out of order — max and + qualify.
  • async let wins for the profile because there are exactly two jobs; a TaskGroup would be ceremony.
  • The Counter needs no lock — the actor is the synchronization. The same loop over a plain class would lose increments to a data race (and won't even compile under Swift 6).
