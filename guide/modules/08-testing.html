Module 08

Unit Testing

The Swift Testing framework end to end: @Suite / @Test / #expect, parameterized cases, testing async code, actors, @Observable models, error paths, and async sequences — with protocol-based test doubles instead of runtime mocking.

Concept

Swift Testing: @Test, #expect, traits

Swift Testing (Xcode 16+) replaces XCTest's subclassing model with free functions and macros. A test is a function annotated @Test; group them in a @Suite struct (a fresh instance per test — natural isolation). Assert with #expect(...) (records a failure, keeps going) or #require(...) (throws to stop the test, and safely unwraps optionals). The expression is captured, so failures show the actual operand values — no XCTAssertEqual zoo.

@Suite("Testing techniques")
struct TestingTests {
    @Test("PriceFormatter formats cents as USD")
    func formatsCurrency() {
        #expect(PriceFormatter().string(fromCents: 150) == "$1.50")
    }
}

Traits configure tests: .disabled("reason"), .tags(.network), .timeLimit(.minutes(1)), .bug("FB123").

Parameterized tests

Pass arguments: to run the same body over many inputs — each case is reported independently, so you see exactly which input failed (vs a single loop that stops at the first failure).

@Test("Formatter across inputs", arguments: [
    (0, "$0.00"), (5, "$0.05"), (199, "$1.99"),
])
func formatsMany(cents: Int, expected: String) {
    #expect(PriceFormatter().string(fromCents: cents) == expected)
}

Testing async/await

A test function can be async — just await the system under test directly. No expectations, no waiters, no timeouts to tune. The runner awaits the test.

@Test func liveServiceFetches() async throws {
    let articles = try await LiveArticleService().fetch()
    #expect(articles.count == 5)
}

Testing actors & @MainActor

Actor-isolated members are reached with await from an async test. To test main-actor code (most SwiftUI types here, given the project's default isolation), annotate the test @MainActor so it runs on the main actor and can touch that state synchronously.

@Test func rateLimiterDenies() async {
    let limiter = RateLimiter(capacity: 2)
    #expect(await limiter.allow())
    #expect(await limiter.allow())
    #expect(await limiter.allow() == false)   // bucket empty
}

Testing @Observable

An @Observable model is just an object: construct it, drive it, assert on its properties. No publisher plumbing. Mark the test @MainActor if the model is main-actor isolated.

Protocol-based test doubles

Swift's preferred substitution is static, not runtime mocking: front a dependency with a protocol, inject a Live implementation in the app and a Fake/Stub in tests. The compiler verifies the double matches the contract, and there's no fragile method-swizzling. (See CatalogService / MockCatalogService in Core/.)

Testing AsyncStream

Drain the sequence with for await into an array, then assert on the collected values and their order.

@Test func pulseStream() async {
    var received: [Int] = []
    for await n in Pulse.stream(count: 4) { received.append(n) }
    #expect(received == [0, 1, 2, 3])
}

Common pitfalls & gotchas

  • Reaching for runtime mocks. Swift has no clean dynamic mocking. Inject protocol-fronted doubles instead of fighting the type system.
  • Sleeping to "wait" for async work. With Swift Testing you await the result directly — a Task.sleep to let things settle is a flaky smell.
  • Locale-dependent assertions. Currency/date formatting varies by locale; inject a fixed Locale (as PriceFormatter does) so assertions are deterministic on CI.
  • Shared mutable state between tests. A @Suite gets a fresh instance per @Test, but global singletons and on-disk stores leak across tests. Use ephemeral UserDefaults / in-memory containers.
  • Missing @MainActor on main-actor SUTs. Touching main-actor state from a non-isolated test won't compile (or forces awaits everywhere). Annotate the test.
  • Asserting on whole objects when you mean one field. Prefer specific #expects so a failure pinpoints the property, not "objects differ."

Challenge

Open SwiftUILabTests/Modules/08-Testing/TestingChallengeTests.swift and fill in the tests for ShippingCalculator and Cart:

  1. Boundary — fee is flat below the free threshold, 0 at and above it.
  2. Parameterized — a single @Test(arguments:) covering several subtotals.
  3. @Observable — Cart add/remove/clear update count and items.
  4. Async/actor — RateLimiter(capacity: 3) allows exactly three times, then denies.

Solution reveal

Show the solution & commentary 
private let calc = ShippingCalculator(freeThresholdCents: 5000, flatFeeCents: 599)

@Test func feeBelowThreshold() { #expect(calc.fee(forSubtotalCents: 4999) == 599) }

@Test func feeAtThreshold() {
    #expect(calc.fee(forSubtotalCents: 5000) == 0)   // boundary is inclusive
    #expect(calc.fee(forSubtotalCents: 9999) == 0)
}

@Test("Fee across subtotals", arguments: [
    (0, 599), (4999, 599), (5000, 0), (12000, 0),
])
func feeMany(subtotal: Int, expected: Int) {
    #expect(calc.fee(forSubtotalCents: subtotal) == expected)
}

@MainActor @Test func cartCount() {
    let cart = Cart()
    cart.add("Apple"); cart.add("Pear")
    #expect(cart.count == 2)
    cart.remove("Apple")
    #expect(cart.items == ["Pear"])
    cart.clear()
    #expect(cart.isEmpty)
}

@Test func limiterCapacity() async {
    let limiter = RateLimiter(capacity: 3)
    #expect(await limiter.allow())
    #expect(await limiter.allow())
    #expect(await limiter.allow())
    #expect(await limiter.allow() == false)
}

Key decisions:

  • The boundary test pins the inclusive behavior (free at 5000), the spot most off-by-one bugs hide.
  • The parameterized case reports each subtotal separately, so a regression names the exact input.
  • The Cart test is @MainActor because the model is main-actor isolated; the limiter test is async because the actor is reached with await.
